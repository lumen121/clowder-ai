# 决策与假设

> 状态：当前基线
> 所属：产品
> 规则效力：已确认决策、默认假设与未决事项
> 维护角色：产品负责人


## 已确认决策

- 产品负责人负责需求、边界、验收标准、风险控制和角色规则。
- 系统架构师负责架构设计和技术决策。
- 在产品文档基线完成并经过架构覆盖性审查前，不应开始实现。
- 首版只支持功能需求和 Bug 修复。
- 首版是本机、单用户、单项目。
- 目标项目是当前 `clowder-ai` 项目。
- 远程仓库是 `git@github-lumen:lumen121/clowder-ai.git`。
- Codex 负责需求统筹、方案设计、任务拆解和整体推进。
- Codex 也可以作为核心开发 Agent。
- Claude 是核心实施和 Bug 修复 Agent。
- MiniMax 是 UI、视觉、多模态、语音、视频、图片和交互体验 Agent。
- 涉及 UI、视觉、多模态、语音、视频、图片或交互体验的任务，MiniMax 必须参与。
- 三个 Agent 都可以在边界和依赖清晰时并行开发。
- 并行开发应在合适情况下使用独立工作区，例如独立分支或 git worktree。
- worktree 隔离可以减少工作区覆盖冲突，但不能消除语义冲突。
- A2A 覆盖完整工作流，不只发生在 Review 阶段。
- Agent 必须评估彼此的计划，不得无脑执行。
- 需求和方案确认前，不得开始开发。
- 除非存在未解决歧义、分歧或需要用户确认的风险，否则 Agent 内部达成一致即可开始开发。
- Bug 修复可以在复现不完整时基于修复假设推进，但必须记录假设、风险和验证路径。
- 作者不能成为自己产出的唯一 Review 方。
- 内部 A2A Review 是必选门禁，不能被 GitHub PR Review 替代。
- 门禁通过后允许推送 feature 分支。
- 直接推送到 `main`、合并到 `main` 和部署都必须人工确认。
- 创建 GitHub PR 是可选能力，需要 GitHub CLI 登录态或 token。
- 产品文档和角色定义只有经过产品负责人确认后才能变更。
- 文档包按 `product/`、`collaboration/`、`agents/`、`architecture/`、`execution/`、`retrospectives/` 分区维护。
- [../00-index.md](../00-index.md) 是文档总入口，负责阅读路由和文档权重说明。
- 复盘文档是历史记录和改进输入，不自动成为当前规则；只有被吸收进当前基线文档后才具备对应规则效力。
- `execution/` 存放执行计划、阶段门禁和执行节奏，不存放构建产物、发布包或最终交付物。
- 暂不创建 `templates/` 目录；模板应在工作项、Review 和复盘闭环实际运行后再沉淀。
- Agent 不得自行选择身份并开始执行；任务分配方或 Harness 必须在启动前明确当前身份、任务目标、任务边界、必读文档、Review 方、验收标准和禁止事项。
- 任务启动上下文应由系统后台生成。对用户而言，这应表现为自动分工和自然协作，而不是要求用户手工填写任务启动包。

## 默认假设

- 本地 Git、账号和 SSH key 已配置。
- 预期 Codex、Claude、MiniMax 的本地 CLI 调用条件已具备，但必须在 `T1 实现基线确认` 中验证。
- 第一轮架构设计由 Codex 以系统架构师模式产出。
- 零阶段由用户人工承担 Harness 职责。
- 系统会逐步自动化零阶段中由人工完成的任务。
- 首版实现应优先保证可追踪性和治理能力，而不是最大化自治程度。

## 未决事项

- 本产品文档包不定义具体架构和技术选型。
- 本产品文档包不定义具体数据模型。
- 本产品文档包不定义具体 UI 布局。
- 本产品文档包不定义具体测试框架和检查命令。
- 是否启用自动 GitHub PR 创建，取决于可用凭证和最终架构设计。
- 首个 Dogfooding 任务应在架构覆盖性审查后选择。
