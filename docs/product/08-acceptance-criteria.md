# 验收标准

> 状态：当前基线
> 所属：产品
> 规则效力：验收基线
> 维护角色：产品负责人


## 产品级验收

首版只有在支持本机、单用户工作流时才算通过验收：用户、Codex、Claude 和 MiniMax 可以围绕功能需求和 Bug 修复进行协作，并且 A2A、Review 门禁、交付门禁和复盘学习均可见、可追踪。

## 功能需求验收

功能需求必须产出：

- 工作项类型：功能需求。
- 目标。
- 范围。
- 已澄清的假设或已解决的歧义。
- 方案。
- 任务拆解。
- 任务负责人、Review 方、依赖和验收标准。
- Agent 在开发前评估过方案的证据。
- 如涉及开发，需有开发产出。
- 非作者 Review 结果。
- 质量门禁结果。
- 在交付范围内时，能提交并推送 feature 分支。

## Bug 修复验收

Bug 修复必须产出：

- 工作项类型：Bug 修复。
- 现象描述。
- 影响范围。
- 可用的复现信息。
- 如复现信息不完整，需有修复假设。
- 风险评估。
- 验证或回归计划。
- 如涉及开发，需有开发产出。
- 非作者 Review 结果。
- 质量门禁结果。
- 在交付范围内时，能提交并推送 feature 分支。

Bug 修复可以基于假设推进，但必须记录假设、风险和验证路径。

## A2A 验收

A2A 只有满足以下条件才算通过：

- A2A 不只出现在 Review 阶段，还出现在澄清、方案设计、任务拆解、执行同步或复盘阶段。
- 至少两个 Agent 围绕同一个工作项发生有意义的交互。
- 关键 A2A 交互记录发起方、接收方、目的、结论和下一步。
- Agent 分歧可见。
- 无法解决的分歧升级给用户。
- 没有 Agent 在不评估的情况下无脑执行另一个 Agent 的计划。

## 开发门禁验收

只有满足以下条件后，才能进入开发：

- 工作项类型已识别。
- 需求歧义已解决或明确记录。
- 方案已形成。
- Agent 已评估方案。
- 没有未解决的内部分歧。
- 任务已定义负责人、边界、依赖、产物、Review 方和验收标准。

## Review 验收

Review 只有满足以下条件才算通过：

- Review 方不是作者本人。
- Review 结论是以下之一：
  - 通过。
  - 需要修改。
  - 存在争议。
  - 需要用户确认。
- 需要修改的问题在交付前已解决。
- 存在争议或需要用户确认的结果不会被静默放行。

当工作影响 UI、视觉、多模态、语音、视频、图片或交互体验时，MiniMax 必须参与方案或 Review。

## 质量门禁验收

质量门禁只有满足以下条件才算通过：

- 检查或验证步骤可见。
- 失败结果可见。
- 系统可以尝试一次安全自动修复。
- 持续失败会使工作项进入阻塞或需要用户确认状态。
- 系统不会隐藏或绕过失败检查。

## 交付验收

交付只有满足以下条件才算通过：

- 内部 Review 已通过。
- 质量门禁已通过，或用户已明确确认风险。
- 可以准备本地提交。
- 可以推送 feature 分支到 `git@github-lumen:lumen121/clowder-ai.git`。
- 系统不会直接推送到 `main`。
- 系统不会自动合并。

GitHub PR 创建是可选能力，取决于 GitHub CLI 登录态或 token 是否可用。

## Dogfooding 验收

Dogfooding 只有满足以下条件才算通过：

- 当工作流具备基本可用性后，Clowder AI 被用于改进 Clowder AI 自身。
- 每个 Dogfooding 工作项记录参与 Agent、人工介入、Review 发现、返工次数、门禁结果、耗时、失败原因和复盘结论。
- 基于真实任务结果提出流程改进。
- 产品规则变更未经产品负责人确认，不会被接受为新基线。
