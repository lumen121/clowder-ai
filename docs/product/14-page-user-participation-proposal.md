# 页面级用户参与入口变更提案

> 状态：已批准
> 所属：产品
> 规则效力：页面级用户参与需求基线
> 维护角色：产品负责人

## 变更背景

现有产品文档已经定义了统一聊天室、时间线、A2A、Harness、Review、门禁和复盘，但没有把“页面级用户参与入口”明确成首版需求。

这会带来两个问题：

- 用户参与路径不清晰，只能依赖 CLI 或零散文本去理解系统。
- 页面、详情、阻塞确认、Review、门禁和复盘虽然被提到，但没有被定义成首版必须提供的可见入口。

本提案的目标不是定义具体 UI 布局，而是明确首版必须提供哪些页面/视图能力，以及用户如何在页面中参与协作闭环。

## 变更目标

首版必须让用户通过页面完成以下事情：

- 录入工作项。
- 查看工作项状态和协作过程。
- 看到 A2A、Review、门禁和复盘结果。
- 在出现歧义、分歧、阻塞或高风险动作时做确认。
- 理解当前任务是谁在做、做到哪一步、为什么卡住。

CLI 可以作为零阶段或内部入口保留，但不能成为首版用户参与的唯一入口。

## 建议新增需求

| ID | WHAT | WHEN | WHY | TradeOff | HOW |
| --- | --- | --- | --- | --- | --- |
| P0-14 | 页面级用户参与入口 | 用户开始一个新工作项、查看协作过程、处理阻塞或确认风险时。 | 让用户能真正参与系统，而不是只能看 CLI 输出。 | 首版页面会多出几个必须维护的入口，但能显著降低参与门槛。 | 提供工作项录入、统一聊天室/时间线、工作项详情、阻塞确认、Review/门禁/复盘可见入口。 |
| P0-15 | 页面级工作项详情与状态可见 | 用户需要理解当前任务边界、负责人、依赖、Review 方和当前状态时。 | 让协作过程从“消息流”变成“可理解的工作流”。 | 需要额外的信息承载，但可减少用户反复追问。 | 页面必须展示目标、范围、任务拆解、当前阶段、阻塞项、待确认项和最近一次关键结论。 |
| P0-16 | 页面级 Review / 门禁 / 复盘查看 | 工作项进入 Review、质量门禁、交付或复盘时。 | 让用户能判断是否可以继续推进、需要修改还是需要自己确认。 | 页面信息更密，但能把风险和结果显性化。 | 页面必须清晰展示 Review 结果、门禁结果、失败原因、复盘结论和改进建议。 |

## 首版页面范围

首版至少需要以下页面/视图能力：

1. 工作项录入入口。
2. 统一聊天室与时间线。
3. 工作项详情视图。
4. 阻塞与人工确认视图。
5. Review、门禁与复盘视图。

## 页面级验收标准

页面能力只有满足以下条件才算可接受：

- 用户可以通过页面提交工作项。
- 用户可以通过页面看见工作项当前状态和最近一次关键协作结论。
- 用户可以通过页面看见 Review、门禁和复盘结果。
- 用户可以在页面中完成歧义确认、分歧确认或风险确认。
- 页面不会把 CLI 输出当作唯一用户入口。
- 页面不要求在本提案阶段定义具体布局、视觉风格或组件技术实现。

## 对现有任务的影响

### T1

不回溯。T1 仍然是环境、Git、CLI、持久化和检查命令的基线确认。

### T2

部分回溯。T2 的 CLI 录入结果可以保留，但需要补充“页面录入入口”要求，不能把 CLI 误认为最终用户入口。

### T3

部分回溯。T3 的持久化基线保留，但需要确认它能否支撑页面查询、详情读取和时间线展示。

### T13

必须重定。T13 不应只是“统一时间线最小 UI”，而应成为首版用户参与主界面的最小实现。

### T16

必须调整。端到端验收必须覆盖页面参与，而不能只验证 CLI 闭环。

## 未决事项

- 页面分成几个视图、是否合并成单页，由架构和设计阶段决定。
- 具体布局、组件、样式和交互细节不在本提案中定义。
- 页面与 CLI 可以并存，但页面是首版主入口，CLI 仅保留为零阶段或内部入口；并存形态由后续架构设计确认。
