# 风险控制

> 状态：当前基线
> 所属：产品
> 规则效力：风险控制规则
> 维护角色：产品负责人


## 必须人工确认的动作

系统必须停止并请求用户确认以下动作：

- 部署。
- 合并到 `main`。
- 直接推送到 `main`。
- 破坏性变更。
- 删除关键文件。
- 绕过失败门禁。
- 跳过 Review。
- Agent 分歧无法解决。
- 需求存在重大歧义。
- 高风险 Bug 缺少验证路径。
- 任何会永久改变外部状态且超出 feature 分支推送的动作。

## 禁止的默认行为

系统不得默认执行：

- 直接推送到 `main`。
- 自动合并 PR。
- 自动部署。
- 隐藏失败结果。
- 跳过内部 A2A Review。
- 在需求和方案确认前开发。
- 未记录的文件修改。
- 用 GitHub PR Review 替代内部 A2A Review。

## Review 控制

- 作者不能成为唯一 Review 方。
- Codex 产出必须由 Claude 或 MiniMax Review。
- Claude 产出必须由 Codex 或 MiniMax Review。
- MiniMax 的实现或产品影响型产出必须由 Codex 或 Claude Review。
- UI、视觉、多模态、语音、视频、图片或交互体验相关工作，MiniMax 必须参与方案或 Review。

## 并行开发控制

只有满足以下条件时才允许并行开发：

- 任务足够独立。
- 边界明确。
- 依赖明确。
- 工作区隔离。
- 合并顺序已知。
- Review 方已指定。
- 存在冲突检测。

出现以下情况时，并行开发必须停止或降级为串行执行：

- 多个 Agent 需要修改同一个核心文件或模块。
- 很可能出现语义冲突。
- 架构方向尚未确定。
- 需求仍有歧义。
- 执行过程中依赖发生变化。

## Bug 修复控制

复现信息不完整时，Bug 修复可以基于修复假设推进。系统必须记录：

- 假设是什么。
- 为什么该假设合理。
- 风险等级。
- 验证方式。
- 剩余不确定性。

高风险 Bug 如果缺少验证路径，必须请求用户确认。

## Git 与交付控制

门禁通过后允许执行：

- 准备本地提交。
- 推送 feature 分支到配置的远程仓库。

必须确认后才能执行：

- 部署。
- 合并到 `main`。
- 直接推送到受保护分支或主干分支。
- 绕过门禁。
- 破坏性清理。

可选执行：

- 在凭证已配置时创建 GitHub PR。

GitHub PR 是外部协作和合并记录，不是内部质量门禁。

## 升级规则

升级给用户时必须包含：

- 发生了什么。
- 哪个规则或门禁阻塞了推进。
- 有哪些选项。
- 每个选项的风险。
- 推荐的下一步。

## Agent CLI 降级规则

如果首周发现 Codex、Claude 或 MiniMax 任一 Agent CLI 不可用：

- 自动化三 Agent 闭环不能算通过。
- 系统不得静默使用模拟结果替代真实 Agent 协作。
- 可以继续采用零阶段人工 Agent 接力作为临时降级方案。
- 降级原因、不可用 Agent、影响范围和恢复条件必须记录为阻塞或执行风险。
- 是否继续推进必须由用户或产品负责人确认。
