# 零阶段复盘记录：产品澄清、架构审查与实现计划审查

> 状态：历史记录
> 所属：复盘
> 规则效力：历史事实、产品侧复盘与改进输入
> 维护角色：产品负责人


## 复盘对象

本复盘记录覆盖 Clowder AI 在自动聊天室和自动 Harness 建成前的零阶段协作过程。

本阶段主要完成：

- 产品需求澄清。
- 产品文档包输出。
- 系统架构设计审查。
- 实现任务拆解与一周开发计划审查。

## 工作项类型

功能需求：建立 Clowder AI 首版产品基线、系统架构基线和开发计划基线。

## 参与角色

- 用户：提出原始目标、持续质疑、确认产品方向和关键取舍。
- 产品负责人：梳理需求、定义边界、输出产品文档、做产品覆盖性审查。
- 系统架构师：基于产品文档输出系统架构设计和实现计划。

## 关键过程

1. 用户提出目标：让 Claude、Codex、MiniMax 三个 Agent 在同一聊天室里协作进行敏捷开发，并通过自学习提升效率和质量。
2. 产品负责人将原始需求澄清为本机、单用户、单项目、首版只支持功能需求和 Bug 修复。
3. 产品负责人明确核心原则：方案先行、A2A 全流程、非作者 Review、Harness 治理、并行 worktree 隔离、人工升级、Dogfooding。
4. 产品负责人输出产品文档包，拆分为 PRD、需求 Backlog、A2A 协议、Harness、Agent 角色、验收标准、风险控制、Dogfooding、零阶段流程和决策假设。
5. 系统架构师输出系统架构设计文档。
6. 产品负责人对架构设计做需求覆盖性审查，并指出表达口径问题：逻辑模型不应被误解为数据库 schema，本地存储表述不应过早暗示事件日志和查询投影技术路线。
7. 系统架构师修订架构文档后，产品负责人确认通过。
8. 系统架构师输出实现任务拆解与一周开发计划。
9. 产品负责人指出开发计划中的覆盖缺口：Agent CLI 适配、工作项录入入口、Git feature 分支交付 P0、worktree 最小治理 P0、任务拆解流程归属。
10. 系统架构师修订开发计划后，产品负责人再次审查。
11. 产品负责人指出 T14 复盘记录不能只是 P1，否则首周最小闭环会削弱自学习目标。
12. 系统架构师将 T14 调整为 P0，将 T15 保留为 P1 Dogfooding 评估增强，产品负责人确认开发计划通过。

## 有效做法

- 用户持续质疑产品负责人输出，帮助避免文档偏实现、A2A 被弱化、Dogfooding 被误解为普通测试。
- 产品负责人明确反驳和修正不合理点，而不是只附和用户或架构师。
- 产品文档先行，使系统架构师有稳定输入，而不是从完整聊天记录中猜最终决策。
- 架构设计经过产品覆盖性审查后再进入实现计划，避免技术方案遗漏产品治理要求。
- 实现计划经过二次审查，把 Agent CLI 接入、工作项录入、worktree 治理、Git 交付和复盘闭环补回 P0。

## 发现的问题

- 早期产品文档曾混入具体技术实现倾向，需要用户提醒后回到产品负责人职责边界。
- 架构师输入提示词一度被写入产品文档，后续确认这类临时操作话术不应沉淀为最终产品结果。
- 开发计划初版更偏流程记录和治理模块，缺少三 Agent 实际接入和工作项录入入口。
- Dogfooding/复盘初版被放在 P1，和“自学习提升效率和质量”的产品目标不完全匹配。
- PowerShell 默认控制台读取中文 Markdown 时出现乱码显示，后续应使用 UTF-8 读取方式核对文档。
- 本地目录曾出现非 Git 仓库状态，这会影响 feature 分支交付、worktree 隔离和首周验收。
- Agent CLI 真实接入是首周关键不确定性，不能用流程记录能力替代三 Agent 实际协作能力。
- 零阶段文档审查门禁和后续系统运行质量门禁需要区分，不能混为同一种质量门禁。

## 人工介入点

- 用户确认产品负责人不负责具体实现设计，只负责产品能力、边界、验收和风险。
- 用户要求 A2A 不只体现于 Review 阶段，而要覆盖完整工作流。
- 用户重新定义三个 Agent 的角色分工。
- 用户要求文档拆分为多份文件，并落盘到 `docs/`。
- 用户要求中文主体输出。
- 用户纠正产品负责人跑偏，要求回到系统架构设计的需求覆盖性审查。
- 产品负责人要求系统架构师修订架构表达口径和实现计划覆盖缺口。

## Review 发现

- 系统架构设计整体覆盖产品要求，但需要说明“数据与状态模型”为逻辑模型，不代表最终数据库 schema。
- “本地事件日志和查询投影的具体存储方案”表述过早暗示技术路线，应改为“本地持久化与查询模型的具体方案”。
- 实现计划初版缺少 Agent CLI 适配层任务。
- 实现计划初版缺少工作项录入入口任务。
- 实现计划初版将 Git feature 分支交付安全流程降为 P1，不符合产品核心交付要求。
- 实现计划初版将 worktree 最小治理降为 P1，不符合并行开发治理要求。
- 实现计划初版没有明确任务拆解、负责人、依赖、Review 方和验收标准的生成流程归属。
- 实现计划初版将复盘记录放在 P1，影响首周最小闭环的自学习能力。

## 返工记录

- 产品交付包从单文档调整为多文档。
- 产品文档从英文主体调整为中文主体，保留必要专有名词。
- 删除产品文档中的临时系统架构师提示词和可复制模板。
- 架构文档补充逻辑模型口径说明。
- 架构文档将存储相关未决项改为技术中性表达。
- 实现计划补充 Agent CLI 适配、工作项录入、任务拆解流程。
- 实现计划调整 Git feature 分支交付和 worktree 最小治理为 P0。
- 实现计划调整复盘记录最小闭环为 P0，Dogfooding 评估增强为 P1。

## 质量门禁结果

- 产品文档包已完成并落盘。
- 系统架构设计已通过产品需求覆盖性审查。
- 实现任务拆解与一周开发计划已通过产品需求覆盖性审查。
- 当前可以进入实现阶段的 `T1 实现基线确认`，不代表可以直接开始功能开发。

## 改进建议

- 后续系统应自动生成类似本复盘的结构化记录，而不是依赖用户和产品负责人手工整理。
- 产品文档、架构文档、实现计划之间应保持显式追踪关系，便于审查覆盖缺口。
- 系统应支持“产品覆盖性审查”这一内建流程，检查架构和计划是否弱化 A2A、Harness、Review、风险控制和 Dogfooding。
- 系统应把“事实、Agent 解释、改进建议、已确认产品规则”分开存储，避免复盘建议被误当成新规则。
- 在中文文档环境下，系统读取和展示 Markdown 时应明确使用 UTF-8。
- 复盘记录应单独分类“技术执行建议”，避免把执行风险或技术建议误读为产品基线。
- Dogfooding 应记录审查反馈如何改变计划，包括初版计划、Review 发现、修改后的计划、任务优先级变化和变更原因。
- Harness 应把实现前置检查固化为进入开发前的固定阶段，而不是依赖人工习惯。
- Harness 应把审查类型作为一等记录，区分产品覆盖性审查、架构执行 Review、实现计划审查、代码 Review 和运行质量门禁。

## 架构侧 Review 吸收结论

- 接受“Git 仓库、远程仓库、SSH key、feature 分支、worktree、未归属变更”作为 `T1 实现基线确认` 的执行风险检查项。
- 接受“Agent CLI 可用性、调用协议、错误处理、超时策略、响应入库方式”作为 `T1` 和 `T4` 的关键执行风险。
- 接受“文档审查门禁”和“系统运行质量门禁”需要分类记录。
- 接受“审查反馈如何改变计划”进入 Dogfooding 记录要求。
- 接受“技术执行建议”作为复盘分类，不自动升级为产品规则。
- UTF-8 问题归类为执行环境风险，不作为独立产品规则。

## 产品负责人确认

- 若首周发现某个 Agent CLI 不可用，自动化三 Agent 闭环不能算通过；可以继续采用人工 Agent 接力推进零阶段，但必须记录为阻塞或降级，不得静默用模拟结果替代。
- Day 7 的 feature 分支交付验收原则上应真实推送到远程；如果 Git 环境或凭证未就绪，只能阶段性接受“交付前检查通过 + 推送阻塞原因可见”，不能视为完整交付能力验收通过。
- “产品覆盖性审查”首版应作为 Dogfooding 和内部治理流程存在，并以审查类型记录；不作为普通用户工作项的完整内建工作流。
- Git 仓库状态、Agent CLI 可用性和 UTF-8 编码问题应补充到本复盘作为执行风险，同时保留架构 Review 文档中的详细分析。
- 后续复盘记录需要单独分类“技术执行建议”，避免被误读为已确认产品规则。

## 已确认产品规则

- 产品负责人负责需求、边界、验收、风险控制和角色规则，不负责具体技术实现。
- 系统架构师负责架构设计、模块设计和技术决策。
- A2A 覆盖完整工作流，不只发生在 Review 阶段。
- 开发前必须方案先行。
- 作者不能是自己产出的唯一 Review 方。
- GitHub PR 不替代内部 A2A Review。
- MiniMax 必须参与 UI、视觉、多模态、语音、视频、图片或交互体验相关任务的方案或 Review。
- 并行开发必须有任务边界、依赖、工作区隔离、Review 方、合并顺序和冲突升级机制。
- Dogfooding 是用系统开发系统自身的实战验证机制，不等同于普通测试。
- 首周最小闭环必须包含结构化复盘记录。
- 复盘记录必须区分事实、解释、改进建议、技术执行建议和已确认产品规则。
- 实现前置检查是进入开发前的固定门禁。
- 审查类型必须可记录、可区分、可追踪。
