## Review 视角

本 Review 不重新评判产品判断是否正确，重点检查零阶段复盘是否会影响后续架构执行和实现落地：

- 是否遗漏对架构执行有影响的信息。
- 是否有表述会误导后续技术设计。
- 是否混淆产品规则、流程建议和技术执行建议。
- 是否有实现风险没有被记录。
- 是否有零阶段流程经验应转成 Harness 或 Dogfooding 的设计要求。
- 是否遗漏对开发计划、worktree、Agent CLI、持久化、Git 交付等执行环节的影响。

本 Review 不自动变更产品规则。需要进入产品基线的内容，应由产品负责人确认。

## 总体结论

[16-phase-zero-retrospective.md](16-phase-zero-retrospective.md) 作为零阶段产品侧复盘是有效的，已经记录了关键过程、人工介入、Review 发现、返工记录、质量门禁结果和已确认产品规则。

从系统架构执行角度看，它不阻塞进入 `T1 实现基线确认`，但不建议把它理解为“可以直接开始功能实现”。下一步仍应严格从 [15-implementation-plan.md](../execution/15-implementation-plan.md) 的 `T1` 开始，先验证项目运行方式、Agent CLI、Git/worktree、持久化和检查命令。

需要补充或澄清的点主要是执行风险和分类口径，不是产品覆盖缺失。

## 架构执行相关遗漏

### 1. Git 仓库与 feature 分支交付风险未显式记录

零阶段过程中曾出现本地 `git status` 返回“not a git repository”的情况。该现象可能只是当前执行环境问题，但它直接影响首版 P0 的 feature 分支交付能力。

建议在执行 `T1 实现基线确认` 时显式验证：

- 当前工作目录是否是 Git 仓库。
- 远程仓库是否配置为 `git@github-lumen:lumen121/clowder-ai.git`。
- SSH key 是否可用。
- 是否能创建 feature 分支。
- 是否能使用 worktree。
- 当前工作区是否存在未归属变更。

这不应被写成产品规则，但应作为实现前置风险记录。

### 2. Agent CLI 可用性和调用协议风险未充分展开

复盘记录中确认了三 Agent 协作目标，但没有记录 Agent CLI 接入的执行风险。开发计划已经补了 `T4 Agent CLI 适配与最小调用闭环`，复盘中也应承认这是首周最关键的不确定性之一。

`T1` 和 `T4` 至少需要确认：

- Codex、Claude、MiniMax 是否都能在本机被调用。
- 每个 Agent 的工作目录、输入格式、输出捕获、错误处理和超时策略。
- Agent 调用失败时是阻塞、降级为人工接力，还是允许临时模拟。
- Agent 响应如何转成 Agent 消息或 A2A 事件。

如果该点不明确，系统可能实现成“流程管理器 + 人工填写 Agent 结果”，而不是三 Agent 协作聊天室。

### 3. “文档审查门禁”和“系统运行质量门禁”需要区分

复盘中“质量门禁结果”主要指产品文档、架构文档和实现计划的审查通过。这是零阶段文档门禁，不等同于后续系统运行时的测试、构建、lint、类型检查或回归验证门禁。

建议后续 Harness 设计中区分：

- 产品覆盖性审查。
- 架构覆盖性审查。
- 实现计划审查。
- 代码/运行质量门禁。
- Git 交付门禁。

否则后续技术设计可能误以为“质量门禁”只有一种。

### 4. 决策来源和版本关系可以记录得更结构化

复盘已经记录了关键返工，但没有显式建立“反馈 -> 修改文档 -> 通过审查”的追踪关系。

建议后续 Dogfooding 记录中结构化记录：

- 谁提出反馈。
- 反馈影响哪份文档。
- 修改后的文档版本或文件路径。
- 是否被产品负责人确认。
- 是否成为产品规则、架构规则或执行建议。

这会直接影响后续复盘记忆能否被 Harness 正确检索和应用。

## 可能误导技术设计的表述

### 1. “当前可以进入开发执行阶段”

该表述合理，但容易被误读成可以跳过准备工作直接实现功能。更准确的执行含义应是：

```text
可以进入实现阶段的 T1 实现基线确认，而不是直接进入功能开发。
```

### 2. “参与角色”中的三 Agent 表述

零阶段中 Codex、Claude、MiniMax 的协作主要是产品和架构层面的角色协作，并不代表系统已经完成 Agent CLI 真实接入。

建议后续避免把零阶段的人工/对话式角色协作误读为系统能力已经存在。系统能力仍需通过 `T4 Agent CLI 适配与最小调用闭环` 验证。

### 3. UTF-8 读取问题属于执行环境风险，不是产品规则

复盘中记录 PowerShell 中文 Markdown 乱码问题是必要的，但它应归类为实现执行注意事项或环境风险，而不是产品规则。

建议后续技术设计把它落到：

- 文件读取编码策略。
- CLI 输出捕获编码策略。
- Markdown 展示编码策略。

## 应补充到 Harness / Dogfooding 的规则

以下建议适合进入 Harness 或 Dogfooding 设计，但是否成为产品规则需产品负责人确认。

### 1. 复盘记录必须区分事实、解释、建议和已确认规则

零阶段已经证明，如果不区分这些类型，复盘建议可能被误认为产品基线。

建议 Harness 在复盘记录中强制区分：

- 观察到的事实。
- Agent 或角色的解释。
- 改进建议。
- 已确认产品规则。
- 技术执行建议。

### 2. 审查类型应成为一等记录

零阶段至少出现了几类审查：

- 产品覆盖性审查。
- 架构执行 Review。
- 实现计划审查。
- 后续代码 Review。

Harness 应记录审查类型、审查人角色、审查对象、结论和是否要求修改。

### 3. 实现前置检查应成为 Harness 的固定阶段

`T1 实现基线确认` 不应只是一次人工习惯动作。后续 Harness 应把它固化为进入开发前的固定检查，包括：

- 项目运行方式。
- Git 仓库状态。
- worktree 可用性。
- 远程仓库配置。
- Agent CLI 可用性。
- 本地持久化路径。
- 测试/构建/检查命令。
- 编码和文件读写策略。

### 4. Dogfooding 应记录“审查反馈如何改变计划”

零阶段最有价值的信息之一，是产品负责人对架构和计划的 Review 直接改变了任务优先级和 P0 范围。

建议 Dogfooding 记录中显式保存：

- 初版计划。
- Review 发现。
- 修改后的计划。
- 被提升或降级的任务。
- 变更原因。

这能帮助后续判断 Review 是否真正提升了交付质量。

## 对后续开发执行的建议

### T1 实现基线确认

T1 应作为硬门禁执行，不应被压缩。建议先验证 Git、Agent CLI、运行命令、持久化路径和编码策略，再开始 T2 之后的开发。

### T4 Agent CLI 适配

Agent CLI 适配应优先实现真实调用闭环。若本地 Claude 或 MiniMax CLI 不可用，应升级给用户或产品负责人确认临时降级策略，不应默认用模拟结果替代。

### T10 Worktree 最小治理

首周不需要完整自动合并，但必须能记录任务到 branch/worktree 的绑定关系，并在合并前检查冲突状态。该能力是并行治理的最低限度。

### T11 Git feature 分支交付

建议先实现交付前检查和 feature 分支准备，再在门禁通过后执行推送。若本地仓库或 SSH 不可用，应记录为阻塞或人工升级，不应静默跳过。

### T14/T15 复盘与 Dogfooding

T14 的结构化复盘记录必须完成，否则 T16 端到端闭环的自学习能力会偏弱。T15 的指标增强可以保持 P1，但建议 Day 6 尽量完成基础指标。

### T16 端到端验证

T16 至少应验证一条成功路径和一个被门禁阻断的失败路径。成功路径证明闭环能跑通，失败路径证明 Harness 不是只记录而不治理。

## 需要产品负责人确认的问题

1. 如果首周发现某个 Agent CLI 不可用，是否允许以“人工 Agent 接力”作为零阶段延续方案，还是必须阻塞首版最小闭环？
2. Day 7 的 feature 分支交付验收，是否必须真实推送到远程，还是允许在凭证或 Git 环境未就绪时以“交付前检查通过 + 推送阻塞原因可见”作为阶段性通过？
3. “产品覆盖性审查”是否要成为 Clowder AI 首版内建工作流，还是仅作为 Dogfooding 和内部治理流程存在？
4. 是否需要将 Git 仓库状态、Agent CLI 可用性和 UTF-8 编码问题补充回 [16-phase-zero-retrospective.md](16-phase-zero-retrospective.md)，还是保留在本架构 Review 文档中即可？
5. 后续复盘记录中，“技术执行建议”是否需要单独分类，避免被误读为已确认产品规则？

## 结论

[16-phase-zero-retrospective.md](16-phase-zero-retrospective.md) 可以作为零阶段复盘基线保留。架构侧建议不要求推翻该复盘，而是补充执行风险、分类边界和 Harness/Dogfooding 可吸收的经验。

在产品负责人确认上述问题前，建议后续开发仍按 [15-implementation-plan.md](../execution/15-implementation-plan.md) 从 `T1 实现基线确认` 开始，不跳过环境、Agent CLI、Git/worktree 和持久化前置验证。

产品负责人已在 [16-phase-zero-retrospective.md](16-phase-zero-retrospective.md) 中吸收并确认本 Review 提出的关键问题。
