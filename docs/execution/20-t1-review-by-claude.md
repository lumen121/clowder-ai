## 三、发现的问题

### 问题 1：证据列缺乏可独立验证的原始输出

Codex 的"证据或命令结果摘要"列使用了叙事性描述（例如"临时 feature 分支创建、切回和删除成功"），而非实际命令输出。Review 方无法仅通过读报告来独立判断 feature 分支、worktree、CLI 调用是否真实执行成功。

**违反规则**：Review 检查项 3 —— "Git 仓库、远程、SSH、feature 分支、worktree 是否都有证据"。

**建议修正**：为 feature 分支创建、worktree 创建/删除、Agent CLI 最小调用补充实际命令和输出日志，使 Review 方可独立验证。

---

### 问题 2：工作区状态陈述不准确（事实偏差）

Codex 声称"工作区干净"，但实际 `git status` 显示：

```text
Changes not staged for commit:
    modified:   docs/execution/00-index.md

Untracked files:
    docs/execution/19-t1-baseline-confirmation-result.md
```

- `00-index.md` 被修改（+1 行，新增 T1 结果文档的索引条目）
- `19-t1-baseline-confirmation-result.md` 是 T1 产出本身，尚未 `git add`

这些变更确实是 T1 执行过程的合理产物，但**不应被描述为"干净"**。Codex 应如实列出这些变更，并注明它们是 T1 自身的产出。

**违反规则**：Review 检查项 2 —— "是否有未验证项被当成已确认事实"。

**建议修正**：将"工作区干净"修正为如实列出 T1 自身产生的变更文件，并注明这些是 T1 执行产出。

---

### 问题 3：SSH 可靠性风险归类不够审慎

Codex 记录了 SSH 连续 3 次失败（全部 attempt 退出码 128），并将其归类为"网络环境波动"和"可降级处理"。但从证据看：

- 3 次重试全部失败，没有一次成功（100% 失败率）
- 报告同时声称远程推送成功——说明后续某次尝试成功了，但未记录成功的那次
- 未记录重试间隔、是否更换网络、是否使用了不同认证方式等排查信息

**影响**：如果 SSH 在 T2-T11 期间频繁不可用，feature 分支推送能力（T11 的核心交付物）将无法可靠验证。

**建议修正**：补充最终推送成功的命令输出日志，说明与 3 次失败的区别（是否更换网络、重试间隔、是否使用了不同认证方式）。

---

### 问题 4：Agent CLI 测试信息不完整

Codex 称"最小调用链已验证"，但未记录：

- 每个 Agent CLI 的具体调用命令和参数
- 实际捕获的 stdout / stderr
- 测试时使用的超时值
- 观察到的实际响应时间
- 错误场景是否被测试（如故意给错误参数）

**违反规则**：Review 检查项 5 —— "Agent CLI 输入、输出捕获、错误处理、超时风险是否记录"——当前仅记录了"需要实现"，而非"已测试并记录结果"。

**建议修正**：补充最小调用的具体命令、参数和输出摘要；如未做错误场景测试，明确标注为 T4 范围。

---

### 问题 5：持久化路径未实际创建或测试

Codex 确认持久化路径为 `data/`，但：

- `data/` 目录实际不存在
- Codex 自己也承认"尚未做业务读写验证"
- 没有最小化的 `writeFile`/`readFile` 测试证据

"确认路径"和"验证读写权限"是两回事。当前只完成了前者。

**建议修正**：创建 `data/` 目录，执行一次最小化的文件写入和读取测试（例如写 `data/.gitkeep` 或一个 JSON 测试文件），将结果记录为证据。

---

### 问题 6：UTF-8 策略确认偏被动

`.gitattributes` 设置了 `* text=auto eol=lf`，但这只覆盖了 Git 层面的换行符处理。Codex 报告中提到的 PowerShell `$OutputEncoding` 风险未被主动测试或验证，仅作为备注。对于中文 Markdown 的 UTF-8 读写，缺少一个最小化的"写中文文件→读回→比对"的测试。

**建议修正**：执行一次最小化 UTF-8 读写往返测试（中文内容写入→读回→比对一致），将结果记录为证据。

## 五、风险判断

| 风险项 | 严重程度 | 说明 |
|---|---|---|
| SSH 间歇性不可用 | **中** | 可能阻塞 T11（feature 分支推送），需在 T2-T10 期间持续观察。当前零阶段由用户人工操作 `git push` 可作为降级方案。 |
| Agent CLI 适配不完整 | **低（T1 阶段）** | T1 只要求确认可调用性，完整的错误处理和超时治理是 T4 的范围。当前证据不足但不改变整体结论。 |
| 持久化未实测 | **低** | `data/` 创建和读写是轻量操作，可在修正中快速补齐。T3 才是正式的持久化实现。 |

| UTF-8 环境依赖 | **低（当前阶段）** | Windows PowerShell 编码问题已知且可控，后续脚本显式设置 UTF-8 即可。 |

**总体风险判断**：当前不存在硬阻塞项。所有问题均可通过文档修正和最少补充操作解决，不需要重做整个 T1。
