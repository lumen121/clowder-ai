# T2 对 P0-14/P0-15/P0-16 的变更影响评估

> 状态：评估完成，等待后续修订任务启动
> 所属：执行
> 规则效力：T2 变更影响评估记录
> 维护角色：Codex（T2 执行 Agent）
> 评估日期：2026-06-19
> 变更来源：`docs/product/14-page-user-participation-proposal.md`（已批准）
> 触发需求：P0-14 / P0-15 / P0-16

## 一、评估结论

当前 T2 产出不满足新的产品基线。

旧版 T2 已完成的是 CLI 工作项录入入口、类型识别、初始记录生成和相关测试。这些结果可以保留为零阶段或内部入口能力，但不能再作为首版用户入口的最终验收基线。

已批准的新基线要求页面级用户参与入口是首版主入口。T2 需要补齐“页面工作项录入 + 创建后初始状态展示/回写”的最小入口；T13 负责后续完整页面主界面，不应把 T13 的完整范围塞进 T2。

## 二、保留

| 保留项 | 涉及产物 | 理由 |
| --- | --- | --- |
| 类型识别逻辑 | `src/work-items/create-work-item.js` | 功能需求 / `bug_fix` 识别、低置信度处理、初始状态生成仍可复用。 |
| CLI 入口 | `bin/clowder-work-item.js` | 可作为零阶段或内部入口保留，但不能作为首版唯一入口。 |
| T2 测试 | `test/work-item-entry.test.js` | 可继续覆盖录入逻辑、类型识别和内部入口行为。 |
| 检查脚本 | `scripts/check-js.js`、`npm run check` | 仍可作为最小质量门禁。 |
| T2 历史交付记录 | `docs/execution/21-t2-work-item-entry-result.md` | 作为旧版执行事实保留，但需要追加变更后状态说明。 |
| T3 Review 结论 | `docs/execution/24-t3-review-by-codex.md` | T3 存储层 Review 仍有效，可支撑后续页面入口落盘。 |

## 三、补齐

| 补齐项 | 建议落点 | 说明 |
| --- | --- | --- |
| 页面工作项录入入口 | T2 修订 | 用户可通过页面提交功能需求或 Bug 修复。页面入口必须真实创建 WorkItem，不能只是展示 CLI 输出。 |
| 页面初始状态展示 | T2 修订 | 创建后展示 ID、类型、初始状态、标题/目标、低置信度提示等最小状态信息。 |
| 页面回写结构化选择 | T2 修订 | 用户显式选择类型或确认类型时，应写入结构化字段，而不是只追加文本。 |
| CLI 边界说明 | T2 文档修订 | 明确 CLI 是零阶段/内部入口，页面是首版主入口。 |
| T2/T3 持久化出口统一 | T2/T3 集成修订 | T2 页面入口应调用 T3 的统一 WorkItem 创建/持久化出口，避免同时存在 `data/work-items/<id>.json` 和 `data/work-items.json` 两套来源。 |
| T2 Review 重新触发 | T2 修订完成后 | 页面入口补齐后，需要非作者 Agent 重新 Review。 |

## 四、回退或改写

| 项目 | 处理方式 | 原因 |
| --- | --- | --- |
| “T2 已满足首版工作项入口”的结论 | 回退 | 当前只有 CLI 入口，不满足页面主入口基线。 |
| “CLI 是 T2 验收入口”的隐含表述 | 改写 | CLI 只能保留为零阶段或内部入口。 |
| T2 文档中的最终状态口径 | 改写 | 旧结果应标记为历史阶段结果，页面补齐后才可作为新基线。 |
| 已完成代码 | 不回退 | 类型识别、CLI、测试仍有复用价值，不需要删除。 |
| T13 完整页面范围 | 不纳入 T2 | T2 只补录入和初始状态，T13 仍负责完整页面主界面、时间线、详情、阻塞确认、Review、门禁和复盘视图。 |

## 五、阻塞

| 阻塞项 | 影响 | 解除条件 |
| --- | --- | --- |
| T2 缺少页面工作项录入入口 | 阻塞 T2 按新基线通过 | 补齐最小页面录入入口，并能真实创建 WorkItem。 |
| T2/T3 持久化出口未统一 | 阻塞 T5 状态机安全启动 | 页面和内部入口统一写入 T3 持久化出口。 |
| 若继续以 CLI 作为首版唯一入口推进 | 与已批准产品基线冲突 | 停止该路径，按页面主入口修订。 |

## 六、需要确认

| 确认项 | 建议 |
| --- | --- |
| T2 最小页面技术边界 | 建议接受极简本地页面/开发服务器入口，只覆盖录入、创建、初始状态展示和类型确认，不扩展到 T13 的完整主界面。 |
| WorkItem 初始状态 | 建议继续使用 `needs_clarification`，除非 T5 状态机另行定义。 |
| T2 特有字段归属 | 需要确认 `raw_request`、`title`、`type_detection` 是否正式进入 T3 WorkItem 模型，或作为扩展字段保存。 |
| MiniMax 是否参与 T2 页面录入 Review | 因涉及页面入口和交互，建议 MiniMax 至少参与方案或 Review；完整主界面仍由 T13 承担。 |

## 七、当前产出是否满足新产品基线

不满足。

当前产出满足旧版“CLI 最小录入入口”的历史目标，但不满足 P0-14/P0-15/P0-16 批准后的首版基线。T2 需要先补齐页面级工作项录入和初始状态展示/回写，再进入新的 T2 通过判定。

## 八、建议的最小修订清单

1. 新增 T2 最小页面录入入口。
2. 页面入口复用 T2 类型识别逻辑。
3. 页面入口通过 T3 统一持久化出口创建 WorkItem。
4. 创建后页面展示初始状态和低置信度提示。
5. 将 CLI 文档和帮助文本改为“零阶段/内部入口”口径。
6. 更新 `21-t2-work-item-entry-result.md`，标记旧结果为历史事实并记录补齐项。
7. 重新运行 `npm run check`、`npm test` 和 T3 相关验证。
8. 触发非作者 Review；涉及页面交互时建议 MiniMax 参与 Review。

## 九、边界说明

本评估不启动新实现，不重做完整方案，不扩大 T2 到 T13 范围。

T13 仍负责完整页面级用户参与主界面，包括统一聊天室/时间线、工作项详情、阻塞确认、Review、门禁、交付和复盘入口。
