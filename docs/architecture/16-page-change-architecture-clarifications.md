`P0-14`、`P0-15`、`P0-16` 已进入产品基线。T2 和 T3 执行 Agent 已输出变更影响评估，产品负责人也已给出产品侧执行澄清：

本文只做架构侧澄清，不改写产品边界。产品验收口径以产品文档为准；实现任务顺序以执行计划为准。

## 总体结论

当前真正的架构硬要求有两个：

- T2 页面入口、CLI 内部入口和后续页面能力必须统一到 T3 的 WorkItem 持久化事实来源。
- 页面查询视角应放在页面层、应用服务层或查询投影中，不能反向污染底层 Store，使 Store 过早承担页面专用逻辑。

其他问题主要是执行边界和验收口径，不需要推翻当前架构。

## T2 最小页面边界

T2 只负责最小页面录入入口，不扩展为完整页面级用户参与主界面。

T2 最小页面入口应覆盖：

- 输入功能需求或 Bug 修复请求。
- 显式选择类型，或使用类型识别结果。
- 真实创建 WorkItem。
- 展示创建后的最小初始状态：ID、类型、状态、标题或目标、低置信度或待确认提示。
- 将用户显式选择或确认写入结构化字段。

T2 不负责：

- 完整统一聊天室与时间线。
- 工作项详情全量视图。
- 阻塞与人工确认完整流程。
- Review、门禁和复盘视图。
- 高保真视觉设计、完整设计系统或复杂路由。

这些仍归 T13 或后续集成任务处理。

## T2 / T3 持久化出口

T2 和 T3 必须形成一个 WorkItem 事实来源。

架构侧要求：

- T2 页面入口应调用 T3 的 `createWorkItem()` 或其上层应用服务。
- CLI 内部入口也应走同一持久化出口。
- `data/work-items/<id>.json` 这类 T2 旧路径只能作为历史产物或临时实现，不得继续作为首版事实来源。
- `data/work-items.json` 或等效 T3 Store 体系应成为 WorkItem 的统一持久化来源。
- 进入 T5 状态机前，必须消除双来源或明确完成迁移。

如果执行层需要保留旧 CLI 行为，应把旧 CLI 改为调用统一应用服务，而不是继续写入独立文件结构。

## T2 特有字段归属

不建议为 T2 单独扩展一套核心 WorkItem 模型。T2 录入字段应映射到 T3 WorkItem 或其 `metadata`：

| T2 字段 | 建议归属 |
| --- | --- |
| `raw_request` | 映射为 `goal`，保留用户原始意图。 |
| `title` | 可由 `goal` 首行派生；如确需保存，可作为可选展示字段。 |
| `type_detection` | 放入 `metadata.type_detection`。 |
| `source` | 放入 `metadata.source`，例如 `page` 或 `cli_internal`。 |

如果当前 T3 工厂尚未显式定义 `metadata` 字段，可以作为扩展字段透传保存，但后续应在模型说明中补齐，避免字段来源不可见。

## 最近一次关键结论

“最近一次关键结论”应由页面查询 / 视图投影层聚合，不要求修改底层 Store。

首版可按以下来源按时间倒序取最近一条会影响用户理解当前状态、下一步动作或风险判断的结构化结论：

- A2AEvent：`conclusion` 或 `next_action`。
- ReviewRecord：`result`、`findings` 或 `required_fixes`。
- QualityGateRun：`result`、`failure_reason` 或 `final_status`。
- EscalationRecord：`recommended_next_step` 或 `user_decision`。
- RetrospectiveMemory：复盘结论或改进建议。

如果当前工作项还没有关键结论，页面显示“待形成”或等价状态即可，不应伪造结论。

## T3 是否必须改代码

架构侧不强制 T3 改代码。

如果现有 `read()`、`list(filter)`、`update()` 已能支撑以下读取和回写，可以不改底层 Store：

- 工作项详情读取。
- 时间线读取。
- 阻塞和待确认项读取。
- Review、门禁、复盘摘要读取。
- 用户确认结果回写。

如果实现时发现页面聚合逻辑重复，可以在页面服务层、应用服务层或查询投影中增加辅助函数。除非存在通用持久化能力缺口，否则不要把页面专用查询直接塞进底层 Store。

## T13 页面形态

首版 T13 可以是一个本地页面或单页多区域界面。

T13 必须覆盖五类能力：

- 工作项录入入口。
- 统一聊天室与时间线。
- 工作项详情视图。
- 阻塞与人工确认视图。
- Review、门禁与复盘视图。

T13 不要求：

- 多页面路由。
- 高保真视觉设计。
- 完整设计系统。
- 复杂导航结构。

如果 T13 范围超过首周时间箱，可以拆成：

- `T13a`：页面级用户参与主界面，覆盖录入、时间线、详情和阻塞确认。
- `T13b`：Review、门禁和复盘视图，覆盖结果查看和失败原因可见。

拆分是执行层降风险手段，不改变 P0 验收目标。

## MiniMax 参与节奏

考虑到 MiniMax 客户端当前操作不友好，允许 T2 最小页面录入入口先由 Codex / Claude 完成。

架构侧接受以下临时执行安排：

- T2 页面入口只按“功能骨架”验收。
- T2 阶段不以完整交互体验作为验收重点。
- MiniMax 可在 T2 最小页面入口搭建完成后再引入。
- MiniMax 必须在 T13 页面主界面验收前，或最迟 T16 E2E 前参与页面体验 Review。
- MiniMax 后续发现的问题，如果影响用户理解、确认或操作效率，应作为 T13 或 T16 修复项处理。

这只是零阶段执行降级，不取消 MiniMax 对 UI、视觉、多模态和交互体验任务的职责。

如果到 T13 或 T16 前 MiniMax 仍不可用，应记录为执行降级或阻塞，不能静默把 MiniMax Review 视为已完成。

## Review 与门禁口径

T2 补齐页面入口后，旧的 CLI 通过结论只能作为历史事实保留。

新的 T2 通过判定必须满足：

- 页面入口真实创建 WorkItem。
- 页面和 CLI 内部入口统一持久化出口。
- 页面展示最小初始状态。
- 相关检查通过。
- 非作者 Review 完成。

T5 状态机启动前，必须确认 WorkItem 事实来源已统一，否则状态机会面对两个来源，风险不可接受。

## 对后续执行的建议

建议后续按以下顺序推进：

1. 补齐 T2 最小页面录入入口。
2. 将 T2 页面入口和 CLI 内部入口统一到 T3 持久化出口。
3. 在 T3 交付说明或补充记录中证明页面查询视角成立。
4. 重新触发 T2 非作者 Review。
5. T5 启动前确认 WorkItem 单一事实来源。
6. T13 实现完整页面级用户参与主界面。
7. T13 或 T16 前引入 MiniMax 做页面体验 Review。

以上建议不替代执行计划，但后续任务启动上下文应引用本文。
