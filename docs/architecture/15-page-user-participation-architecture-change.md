# 页面级用户参与入口：架构影响评估与变更包

> 状态：草案
> 所属：架构
> 规则效力：页面级用户参与需求的架构影响评估与变更建议
> 维护角色：系统架构师


## Review 对象

- [../product/14-page-user-participation-proposal.md](../product/14-page-user-participation-proposal.md)
- [../product/02-requirements-backlog.md](../product/02-requirements-backlog.md)
- [14-system-architecture-design.md](14-system-architecture-design.md)
- [../execution/15-implementation-plan.md](../execution/15-implementation-plan.md)

本变更包用于评估产品侧新增的页面级用户参与需求对当前架构基线和执行计划的影响。

## 总体结论

这次产品侧变更是合理且必要的。它没有推翻现有系统架构方向，但会改变首版“用户入口”和“最小 UI”的定义：

- 现有架构仍然成立，不需要从头重做。
- 当前架构与执行计划的主要缺口不在核心治理逻辑，而在用户参与表面的定义过弱。
- `T1` 不应回溯，可按原计划继续。
- 在 `P0-14`、`P0-15`、`P0-16` 获得产品负责人正式批准后，必须更新：
  - [14-system-architecture-design.md](14-system-architecture-design.md)
  - [../execution/15-implementation-plan.md](../execution/15-implementation-plan.md)
- 在批准前，本变更包只作为架构建议，不覆盖当前基线。

一句话结论：

```text
当前架构可以吸收这次变更，但必须把“页面级用户参与”提升为首版一等能力，而不是把 T13 继续当成时间线可视化附属物。
```

## 影响级别

### 高影响

- 首版用户入口定义
- 页面层和时间线/详情视图设计
- `T13` 最小 UI 任务范围
- `T16` 端到端验收路径

### 中影响

- `T2` 工作项录入入口
- `T3` 本地持久化对页面查询与详情展示的支撑
- `T12` 人工升级和阻塞确认的页面承载方式
- Review、门禁、复盘结果的页面呈现方式

### 低影响

- A2A 协议本身
- worktree 治理本身
- Git feature 分支交付本身
- Harness 的“治理层”定位

## 架构影响分析

### 1. 用户参与表面需要成为首版显式组件

当前架构已经有：

- 统一聊天室与时间线
- 工作项控制器
- A2A 编排器
- Harness
- Review / 门禁 / 复盘记录

但这些主要是“治理闭环”视角，不足以回答产品新要求中的关键问题：

- 用户从哪里创建工作项。
- 用户从哪里看工作项详情。
- 用户从哪里完成阻塞确认或风险确认。
- 用户从哪里看 Review、门禁和复盘结果。

因此，架构上应新增或显式化一个组件层：

```text
页面级用户参与层
```

它不是一个独立的新业务域，而是用户访问现有治理能力的主要表面。

建议它至少包含以下视图能力：

- 工作项录入入口
- 统一聊天室与时间线
- 工作项详情视图
- 阻塞与人工确认视图
- Review / 门禁 / 复盘视图

### 2. 页面需求不会推翻核心领域模型，但会增强查询模型要求

当前逻辑模型中的这些实体仍然有效：

- `WorkItem`
- `Task`
- `A2AEvent`
- `ReviewRecord`
- `QualityGateRun`
- `EscalationRecord`
- `RetrospectiveMemory`

问题不在“缺实体”，而在“缺面向页面的读取视角”。

页面级需求会要求至少具备以下查询能力：

- 工作项摘要查询
- 最近一次关键结论查询
- 当前阻塞项和待确认项查询
- 当前负责人、Review 方和依赖查询
- Review 结果汇总查询
- 门禁结果汇总查询
- 复盘结果摘要查询

因此，`T3` 需要确认的不只是“能存”，还要确认“能查给页面看”。

这不要求现在就引入复杂的 CQRS 或事件溯源，只要求：

- 写入模型和读取模型不要耦合死在纯消息流里
- 页面详情和时间线能基于结构化数据读取

### 3. 时间线不能再作为唯一用户可见表面

当前 `T13` 的定义仍偏向：

```text
统一时间线最小 UI
```

这在旧基线下够用，但在新提案下不够。

页面级用户参与要求的是：

- 时间线只是其中一个视图
- 用户还需要详情、确认、Review/门禁/复盘查看能力

因此，架构上应把时间线视为：

```text
页面级用户参与层中的一个核心视图，而不是全部
```

### 4. 人工升级记录需要有“可操作页面入口”

现有架构中 `EscalationRecord` 已经足以记录：

- 发生了什么
- 阻塞规则
- 可选项
- 风险
- 推荐动作
- 用户决定

但新提案要求用户“通过页面完成确认”，所以光有记录还不够，必须有可见、可操作入口。

这意味着：

- `T12` 不能只生成升级记录
- 页面层必须能消费升级记录
- 用户确认结果必须回写为结构化状态，而不是只停留在聊天文本

### 5. Review / 门禁 / 复盘结果必须从“内部记录”变成“页面结果”

当前 `T9`、`T14`、`T15` 更多是结果记录与治理逻辑。

产品侧变更后，架构必须明确：

- Review 结果不仅要存在，还要能被页面直接展示
- 门禁失败原因不仅要记录，还要能被用户判断“是继续、修改还是确认风险”
- 复盘不仅要沉淀到记忆，还要有用户查看入口

因此，页面层与查询层的关系变得更重要，但核心治理逻辑无需重做。

## 对当前架构文档的建议调整

若 `P0-14`、`P0-15`、`P0-16` 被正式批准，建议对 [14-system-architecture-design.md](14-system-architecture-design.md) 做以下更新：

### 1. 在主要组件中新增或显式化

新增组件：

```text
页面级用户参与层
```

职责：

- 提供工作项录入页面入口
- 提供统一聊天室与时间线视图
- 提供工作项详情视图
- 提供阻塞与人工确认视图
- 提供 Review / 门禁 / 复盘视图

### 2. 在数据与状态模型中补充读取视角说明

不一定新增源实体，但应补充说明：

- 页面详情查询
- 时间线查询
- 阻塞与待确认项查询
- Review / 门禁 / 复盘摘要查询

### 3. 在核心流程中补充页面参与路径

例如：

- 用户通过页面提交工作项
- 用户通过页面查看当前状态和关键结论
- 用户通过页面处理阻塞确认和风险确认
- 用户通过页面查看 Review / 门禁 / 复盘结果

### 4. 在产品需求覆盖表中增加映射

新增映射：

- 页面级用户参与入口
- 页面级工作项详情与状态可见
- 页面级 Review / 门禁 / 复盘查看

## 对执行计划的影响

### T1

不回溯。

理由：

- `T1` 是环境与执行前置门禁
- 页面级需求不改变 Git、CLI、worktree、持久化、检查命令和编码策略的验证范围

### T2

部分回溯，必须扩展。

当前 `T2` 只保证“工作项录入与类型选择/识别最小入口”。在新提案下，不能再把 CLI 录入视为最终用户入口。

建议更新方向：

- `T2` 明确包含页面录入入口
- CLI 录入保留为零阶段或内部入口
- 页面录入成为首版用户入口基线

### T3

部分回溯，必须增强。

当前 `T3` 强调本地读写。新提案要求它还要支撑：

- 页面详情读取
- 时间线读取
- Review / 门禁 / 复盘结果读取
- 阻塞与待确认项读取

建议更新方向：

- 在 `T3` 验收中加入“能支撑页面查询”的要求

### T12

需要增强，但不必重写任务边界。

当前 `T12` 是人工升级流程。新提案要求阻塞和风险确认有页面入口，因此：

- `T12` 仍负责结构化升级记录
- 页面层负责呈现和回写确认动作

建议在执行计划中明确这种分工。

### T13

必须重定。

当前 `T13` 定义为：

```text
统一时间线最小 UI
```

在新提案下应调整为：

```text
页面级用户参与主界面的最小实现
```

至少覆盖：

- 工作项录入入口
- 统一聊天室与时间线
- 工作项详情
- 阻塞与人工确认
- Review / 门禁 / 复盘可见入口

如果维持一周计划不变，`T13` 的范围会明显扩大。

建议在执行计划修订时明确两种方案：

#### 方案 A：维持单任务

- 直接扩大 `T13`
- 风险是 UI 范围膨胀，Day 5~Day 7 压力上升

#### 方案 B：拆成两个并行 UI 子任务

- `T13a` 页面级用户参与主界面：录入、时间线、详情、确认
- `T13b` Review / 门禁 / 复盘视图

如保持一周时间箱不变，执行计划可考虑方案 B，以降低单个 UI 任务范围膨胀风险。是否调整任务编号和计划结构，应由执行计划修订决定。

### T14

逻辑不变，但其输出必须支持页面查看。

即：

- `T14` 仍是复盘记录最小闭环
- 新要求是这些结果必须能被页面层消费

### T16

必须调整。

当前 `T16` 验收要求已经覆盖录入、Agent 协作、门禁、Review、交付和复盘，但没有明确“页面参与路径”。

建议调整为至少覆盖：

- 用户通过页面录入工作项
- 用户通过页面查看状态和关键结论
- 用户通过页面处理一次阻塞或风险确认
- 用户通过页面查看 Review / 门禁 / 复盘结果

同时仍需保留：

- 一条成功路径
- 一条被门禁阻断的失败路径

## 对后续 Agent 执行阅读路径的影响

这次变更不会改变 Agent 的阅读基线权重，但会改变以下执行习惯：

- `T2`、`T3`、`T13`、`T16` 的执行 Agent 需要把页面级用户参与提案作为任务上下文的一部分
- 页面相关任务建议优先由 MiniMax 参与设计或 Review
- Codex 在任务拆解时，不能再把“页面”仅理解为时间线展示层

建议在提案批准后，把 [../product/14-page-user-participation-proposal.md](../product/14-page-user-participation-proposal.md) 加入：

- `T2`
- `T3`
- `T13`
- `T16`

的任务启动上下文必读材料。

## 风险

### 1. T13 范围膨胀风险

如果不拆分 UI 子任务，首周计划的 UI 负担会明显上升。

### 2. 页面需求被误实现为“高保真设计任务”

本提案定义的是页面级参与能力，不是完整视觉设计方案。若后续把重点放到布局、视觉和组件细节，会稀释治理闭环目标。

### 3. 录入入口双轨不清风险

若不明确页面入口和 CLI 的优先级，可能出现：

- CLI 是真实入口
- 页面只是演示入口

这会违背产品新提案的初衷。

### 4. 查询模型不足风险

如果 T3 只做写入，不做页面所需读取整理，T13 会被迫从原始消息流和分散记录中硬拼页面，导致实现复杂度上升。

## 建议结论

1. 当前不回滚 `T1`，继续执行实现前置检查。
2. 在产品负责人正式批准 `P0-14`、`P0-15`、`P0-16` 前，不直接改写当前基线架构文档和执行计划。
3. 一旦批准，优先修订：
   - [14-system-architecture-design.md](14-system-architecture-design.md)
   - [../execution/15-implementation-plan.md](../execution/15-implementation-plan.md)
4. 执行计划修订的重点不是“多做一个页面”，而是把页面级用户参与提升为首版主路径。
5. 如果保持首周时间箱不变，可考虑将 `T13` 拆成两个并行子任务，降低 UI 范围膨胀风险。

## 需要产品负责人确认的问题

1. `P0-14`、`P0-15`、`P0-16` 是否会在本轮变更后直接进入 `approved`，还是继续保持 `proposed` 待确认？
2. 首版页面级用户参与是否要求“页面为主、CLI 为辅”的单一路径，还是允许页面和 CLI 作为同等一级入口并存？
3. 若保持一周开发节奏不变，是否接受将 `T13` 拆分为两个 UI 子任务以控制风险？
4. 页面中的阻塞确认和风险确认，是否只要求“确认动作”，还是允许用户直接在页面改写任务边界、负责人和验收标准？

## 结论

页面级用户参与入口是首版用户体验和治理闭环的一次重要纠偏，不是对现有架构的否定。

当前架构可以吸收这次变更，但吸收方式应是：

- 保留现有治理核心
- 增强页面级用户参与层
- 提升查询与详情视图能力
- 重定 `T13`
- 调整 `T16`

只要在产品批准后及时修订架构和执行计划，这次变更不会阻塞 `T1`，但会显著影响 `T2` 之后的执行路径。
