# Harness 治理要求

> 状态：当前基线
> 所属：协作治理
> 规则效力：Harness 治理规则
> 维护角色：产品负责人


## 目的

Harness 是让多 Agent 工作保持可控、可观测、可恢复和安全的治理层。它不只是工具调用器，还必须管理 Agent 行为周围的上下文、记忆、权限、状态、护栏和反馈回路。

业界并没有一个统一命名为 Harness 的硬标准，但成熟 Agent 系统通常会收敛到本文档列出的能力。

## 必需的 Harness 能力

### 上下文管理

Harness 必须确保 Agent 获得相关的工作项上下文、历史决策、角色指令、约束、当前状态和任务输入。

当上下文陈旧或不完整会带来风险时，Harness 必须阻止 Agent 继续操作。

### 记忆管理

Harness 必须存储和检索本地复盘记忆：

- 历史工作项结果。
- Review 发现。
- 失败原因。
- 有效协作模式。
- 用户介入原因。
- 质量门禁结果。
- 流程改进决策。

记忆可以影响后续工作，但不能覆盖当前用户指令或明确的产品规则。

### 工具沙箱

Harness 必须控制工具访问权限和工作区范围。Agent 应只获得完成当前任务所需的工具和工作区域。

高风险工具使用必须经过门禁或升级。

### 权限管理

Harness 必须区分：

- 只读动作。
- 计划动作。
- 文件修改。
- 测试或检查执行。
- 提交准备。
- Feature 分支推送。
- PR 创建。
- 部署。
- 合并主干。

部署、合并主干、破坏性动作和绕过门禁必须人工确认。

### Agent Loop 控制

Harness 必须防止失控循环。它应管理：

- 当前状态。
- 下一步允许动作。
- 重试次数上限。
- Review 循环。
- 升级条件。
- 停止条件。

### 校验护栏

Harness 必须强制执行：

- 方案先行。
- 实现前置检查。
- 非作者 Review。
- 质量门禁。
- Review 结果处理。
- 必要的用户确认点。
- 失败和分歧可见。

实现前置检查应至少覆盖：

- 项目运行方式。
- Git 仓库状态。
- 远程仓库配置。
- worktree 可用性。
- Agent CLI 可用性。
- 本地持久化路径。
- 测试、构建或检查命令。
- 文件读写和编码策略。

### 安全约束

Harness 必须防止默认执行：

- 直接推送到 `main`。
- 自动合并 PR。
- 隐藏失败并继续。
- 未经 Review 的交付。
- 未记录的项目状态变更。
- 需求和方案未确认就开发。

### 任务路由与编排

Harness 必须基于以下因素路由任务：

- 工作项类型。
- Agent 角色。
- 任务边界。
- 依赖关系。
- 是否属于 MiniMax 必须参与的任务类别。
- 当前工作负载。
- 风险等级。
- Review 独立性。

Harness 必须在启动 Agent 前生成任务启动上下文。任务启动上下文至少包含：

- 当前身份。
- 任务 ID。
- 任务目标。
- 任务边界。
- 依赖。
- 必读文档。
- Review 方。
- 验收标准。
- 禁止事项。

Agent 不得自行选择身份、职责或 Review 角色并开始执行。若任务启动上下文缺失或身份不明确，必须暂停并升级给任务分配方或用户确认。

用户体验上，任务启动上下文应由系统后台自动生成。用户只需要提交工作项；只有当类型、边界、分工或风险存在歧义时，系统才应要求用户确认。

### 共享状态管理

Harness 必须维护共享工作项状态，并让状态变化可见。状态至少应包含当前阶段、负责人、阻塞项、待决策问题、Review 状态、质量状态、交付状态和复盘状态。

### 人工升级

出现以下情况时，Harness 必须升级给用户：

- 需求歧义阻塞推进。
- Agent 无法解决重大分歧。
- 高风险 Bug 缺少验证路径。
- 质量门禁失败且无法安全修复。
- 请求或暗示破坏性动作。
- 请求部署或合并主干。
- 请求绕过门禁。

### 工作区与产物治理

并行开发时，Harness 应支持隔离工作区，例如独立分支或 git worktree。

Harness 必须跟踪：

- 哪个 Agent 拥有哪些工作区。
- 哪个任务对应哪个分支或 worktree。
- 发生了哪些变更。
- 合并顺序。
- 冲突状态。
- 清理和归档状态。

### 观测与评估

Harness 必须记录：

- 工作项类型。
- 参与 Agent。
- A2A 交互。
- 决策和分歧。
- 审查类型。
- 审查人角色。
- 审查对象。
- 审查结论。
- 审查是否要求修改。
- 人工介入。
- Review 发现。
- 返工次数。
- 门禁结果。
- 耗时。
- 失败原因。
- 交付结果。

### 反馈回路

Harness 必须将复盘发现反馈到后续工作中。改进可能影响：

- 角色指令。
- 任务路由。
- Review 规则。
- 澄清问题。
- 质量门禁。
- 风险控制。

产品规则和角色定义只有经过产品负责人确认后，才能更新为新的基线。
