# A2A 协作协议

> 状态：当前基线
> 所属：协作治理
> 规则效力：A2A 协作规则
> 维护角色：产品负责人


## 目的

A2A 协议定义 Codex、Claude、MiniMax 之间如何协作。这是产品需求，不是技术实现方案。

目标是避免系统变成三个互不相关的聊天机器人。Agent 必须能够互相评估、质询、纠错、Review 和协同推进。

## 原则

- A2A 覆盖完整工作流，不只发生在 Review 阶段。
- Codex 主导需求统筹，但 Claude 和 MiniMax 必须评估 Codex 的分析和任务拆解。
- Agent 不得无脑执行其他 Agent 的计划。
- 需求和方案没有充分确认前，不得进入开发。
- 内部分歧必须可见。
- 无法解决的分歧必须升级给用户。
- 作者不能成为自己产出的唯一 Review 方。

## A2A 交互类型

- 澄清请求。
- 需求质询。
- 方案问题。
- 风险提示。
- 任务拆解反馈。
- 任务交接。
- 执行同步。
- Review 请求。
- 修改要求。
- 验证请求。
- 分歧升级。
- 复盘反馈。

## 关键 A2A 记录字段

每次关键 A2A 交互必须记录：

- 发起 Agent。
- 接收 Agent。
- 工作项 ID 或任务 ID。
- 目的。
- 上下文。
- 问题、主张或请求动作。
- 响应。
- 结论。
- 下一步动作。
- 是否需要用户介入。

## 标准 A2A 流程

### 1. 类型识别与澄清

Codex 提出工作项类型。Claude 和 MiniMax 可以在用户输入暗示不同类型或信息不足时提出质疑。

首版支持的类型：

- 功能需求。
- Bug 修复。

### 2. 需求澄清

Agent 协同识别缺失信息。如果缺失信息会阻塞安全推进，系统应询问用户。如果可以基于假设继续推进，必须记录假设。

Bug 修复在复现信息不完整时，可以基于修复假设推进，但必须记录假设、风险和验证路径。

### 3. 方案设计

Codex 主导方案设计。Claude 和 MiniMax 必须评估：

- 方案是否符合需求。
- 方案是否可测试。
- 是否存在隐藏风险。
- 任务拆解是否合理。
- 是否因为涉及 UI、视觉、多模态、语音、视频、图片或交互体验而需要 MiniMax 参与。

### 4. 任务拆解

每个任务必须定义：

- 负责人。
- 协作者。
- 边界。
- 依赖。
- 预期产物。
- Review 方。
- 验收标准。
- 是否适合并行开发。

### 5. 开发同步

开发过程中，出现以下情况时 Agent 必须同步：

- 依赖发生变化。
- 任务边界变得不清晰。
- 任务触及共享区域。
- 假设发生变化。
- 质量门禁失败。
- 可能出现合并冲突或语义冲突。

### 6. 交叉 Review

提交或推送前，作者必须请求非作者 Agent Review。

Review 结果必须是以下之一：

- 通过。
- 需要修改。
- 存在争议。
- 需要用户确认。

### 7. 复盘反馈

交付或失败后，Agent 需要记录：

- 哪些做法有效。
- 哪里失败了。
- 用户在哪里介入。
- A2A 在哪里提升了质量。
- A2A 在哪里造成了延迟或混乱。
- 哪些规则需要改进。

## 并行开发规则

只有满足以下条件时，Agent 才能并行开发：

- 任务边界清晰。
- 依赖明确。
- 产物可独立 Review。
- Review 方已指定。
- 合并顺序已知。
- 存在工作区隔离，例如独立分支或 git worktree。
- 存在冲突检测和升级路径。

出现以下情况时，不应并行开发：

- 多个 Agent 很可能修改同一个核心文件或模块。
- 架构方向尚未确定。
- 需求仍存在重大歧义。
- 质量门禁未定义。
- 合并顺序不清晰。
