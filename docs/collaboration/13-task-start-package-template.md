# 任务启动包模板

> 状态：产品侧治理基线，已通过系统架构师轻量复核，待架构侧执行落地
> 所属：协作治理
> 规则效力：任务启动上下文的产品侧模板与阻断规则
> 维护角色：产品负责人

## 目的

任务启动包用于在 Agent 开始执行前，明确身份、目标、边界、依赖、上下文、Review 方和验收规则。

它不是最终数据库 schema，也不限定具体技术实现。系统架构师可以将它落成文档、表单、结构化记录、页面输入或 Harness 内部对象，但不得降低其中的启动门禁和产品治理要求。

## 使用场景

以下场景必须提供任务启动包：

- 新功能需求执行。
- Bug 修复执行。
- 页面、UI、视觉、多模态、语音、视频、图片或交互体验任务。
- Review 任务。
- 质量门禁或交付检查任务。
- Dogfooding、复盘或流程改进任务。
- 零阶段人工分发给 Codex、Claude、MiniMax 的任务。
- 未来 Harness 自动调度给 Agent 的任务。

## 总体规则

- Agent 不得自行选择身份并开始执行。
- 任务启动包必须由任务分配方或 Harness 生成。
- Agent 只能按照启动包指定身份执行。
- Agent 默认只能执行启动包明确列出的允许动作。
- 启动包字段缺失、互相冲突或与当前产品/架构/执行基线冲突时，Agent 必须暂停并请求补齐或升级确认。
- 启动包不能绕过产品基线、架构约束、执行计划、A2A Review、质量门禁或人工确认规则。
- 启动包中的实现方式建议不能反向覆盖产品需求。

## 标准模板

### 1. 基本信息

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 任务 ID | 是 | 对应执行计划、工作项或临时任务编号。 | 阻断执行。 |
| 任务类型 | 是 | 功能需求、Bug 修复、Review、门禁检查、复盘、文档治理等。首版产品工作项只支持功能需求和 Bug 修复。 | 阻断执行。 |
| 当前身份 | 是 | 明确本次执行身份，如 Codex、Claude、MiniMax、系统架构师。 | 阻断执行。 |
| 任务来源 | 是 | 用户需求、执行计划、变更包、Review 结论、复盘改进项等。 | 阻断执行。 |
| 关联工作项 | 条件必填 | 如果任务来自某个 WorkItem，必须填写。 | 有关联工作项但未填写时阻断执行。 |
| 优先级 | 是 | P0、P1、P2 或执行计划中的等价优先级。 | 阻断执行。 |

### 2. 目标与边界

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 任务目标 | 是 | 本次任务要完成什么。 | 阻断执行。 |
| 范围内事项 | 是 | 本次允许处理的内容。 | 阻断执行。 |
| 范围外事项 | 是 | 本次明确不处理的内容，防止范围膨胀。 | 阻断执行。 |
| 交付物 | 是 | 预期产出，如代码、文档、Review 记录、验证记录、页面入口等。 | 阻断执行。 |
| 完成标准 | 是 | 什么情况下算本任务完成。 | 阻断执行。 |

### 3. 上下文与必读材料

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 必读文档 | 是 | 本次任务开始前必须读取的文档清单。 | 阻断执行。 |
| 当前任务上下文 | 是 | 与本任务直接相关的背景、当前状态、已完成内容、已知限制。 | 阻断执行。 |
| 依赖任务 | 条件必填 | 如果依赖其他任务、模块或文档，必须列明。 | 有依赖但未填写时阻断执行。 |
| 已知变更影响 | 条件必填 | 如果任务受变更包影响，必须说明影响范围。 | 有变更影响但未填写时阻断执行。 |
| 默认假设 | 条件必填 | 如果允许带假设推进，必须明确假设和验证方式。 | 使用假设但未填写时阻断执行。 |

页面相关任务还必须包含：

- [15-page-change-implementation-clarifications.md](../product/15-page-change-implementation-clarifications.md)

## 4. 分工与协作

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 主执行 Agent | 是 | 本次任务主要执行者。 | 阻断执行。 |
| 协作 Agent | 条件必填 | 有 A2A 协作、页面体验、多模态或跨模块任务时必须填写。 | 需要协作但未填写时阻断执行。 |
| Review 方 | 是 | 非作者 Review 方。作者不能作为自己产出的唯一 Review 方。 | 阻断执行。 |
| A2A 协作要求 | 条件必填 | 需要澄清、方案评估、拆解反馈、执行同步、Review 或复盘时填写。 | 需要 A2A 但未填写时阻断执行。 |
| MiniMax 参与要求 | 条件必填 | 涉及 UI、视觉、多模态、语音、视频、图片或交互体验时必须填写。 | 相关任务未填写时阻断执行。 |

MiniMax 参与规则：

- 涉及 UI、视觉、多模态、语音、视频、图片或交互体验的任务，MiniMax 必须参与方案或 Review。
- 零阶段 T2 最小页面录入可先由 Codex 或 Claude 完成和 Review。
- MiniMax 必须在 T13 页面主界面或 T16 E2E 前参与页面体验 Review。

## 5. 执行约束

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 允许动作级别 | 是 | 明确本任务允许 Agent 做到哪一级动作，如只读、方案/计划、文件修改、检查执行、提交准备、feature 分支推送、PR 创建、部署、合并主干。 | 阻断执行。 |
| 禁止事项 | 是 | 本任务不得做的动作，如不得扩大范围、不得绕过 Review、不得直接推 main。 | 阻断执行。 |
| 文件/模块边界 | 条件必填 | 开发任务必须说明主要允许触碰的区域，或说明由执行 Agent 先识别后汇报。 | 开发任务缺失时阻断执行。 |
| 维护性注释要求 | 条件必填 | 开发任务必须说明是否需要补充维护性注释及口径。 | 开发任务缺失时阻断执行。 |
| Git 身份要求 | 条件必填 | 涉及 Git 写入动作时必须明确 user.name 和 user.email 口径。 | 涉及 Git 写入但缺失时阻断执行。 |
| worktree / 分支要求 | 条件必填 | 并行开发或隔离执行时必须说明。 | 需要隔离但缺失时阻断执行。 |
| 高风险动作限制 | 是 | 是否涉及部署、合并主干、破坏性操作、绕过门禁等。 | 阻断执行。 |

允许动作级别应按最小权限原则填写。可选层级包括：

- `只读`：只能阅读文档、代码、日志或状态，不能修改文件。
- `方案/计划`：可以输出方案、任务拆解、Review 意见或计划，不修改实现文件。
- `文件修改`：可以修改启动包范围内的文件。
- `检查执行`：可以运行启动包允许的测试、lint、构建、页面验证或检查命令。
- `提交准备`：可以整理变更、准备 commit 信息或交付说明，但不推送。
- `feature 分支推送`：可以推送指定 feature 分支。
- `PR 创建`：可以创建 PR，但不能自动合并。
- `部署`：涉及部署动作，必须人工确认。
- `合并主干`：涉及合并到 `main` 或等价主干，必须人工确认。

部署、合并主干、破坏性文件操作、绕过 Review、绕过质量门禁、自动合并 PR 必须显式人工确认；未被启动包列为允许动作的行为默认禁止。

## 6. 验收与门禁

| 字段 | 必填 | 说明 | 缺失处理 |
| --- | --- | --- | --- |
| 验收标准 | 是 | 本任务必须满足的验收条件。 | 阻断执行。 |
| 验证方式 | 是 | 需要运行的检查、测试、页面验证、文档 Review 或人工确认。 | 阻断执行。 |
| Review 通过标准 | 是 | Review 需要判断什么、输出什么结论。 | 阻断执行。 |
| 质量门禁 | 条件必填 | 涉及代码、页面、流程推进或交付时必须填写。 | 相关任务缺失时阻断执行。 |
| 失败处理 | 是 | 验收或门禁失败时如何记录、回退、升级或重做。 | 阻断执行。 |

## 7. 人工升级条件

任务启动包必须明确以下情况是否需要升级给用户或产品负责人：

- 需求存在歧义，且 Agent 内部无法达成一致。
- 产品规则、架构设计、执行计划之间存在冲突。
- 任务目标与启动包边界冲突。
- 需要扩大 P0 范围或降低 P0 验收标准。
- 需要绕过 Review、质量门禁或安全限制。
- 需要执行高风险 Git 操作、部署、破坏性文件操作或合并主干。
- 页面用户参与路径无法满足已批准的 P0-14、P0-15、P0-16。
- MiniMax 应参与但当前无法参与，且任务涉及用户体验判断。

## 8. 输出记录要求

Agent 完成任务后，至少要输出：

- 实际完成内容。
- 未完成内容或偏离项。
- 修改过的文件或文档。
- 验证方式和结果。
- Review 结论或待 Review 状态。
- 建议下一状态或交接状态。
- 是否解除依赖。
- 是否阻断后续任务。
- 是否触发 Review、质量门禁、人工确认或复盘。
- 后续阻断关系和被影响任务。
- 遗留风险。
- 是否需要用户、产品负责人或系统架构师确认。

关键结论不能只留在对话上下文里，必须进入对应文档、执行记录、Review 记录或复盘记录。

## 按任务类型的补充要求

### 功能需求

必须补充：

- 用户目标。
- 业务范围。
- 方案先行结果。
- 任务拆解结果。
- 非作者 Review 方。
- 页面可见性要求。

### Bug 修复

必须补充：

- Bug 表现。
- 复现条件。
- 影响范围。
- 修复假设。
- 验证方式。
- 是否需要回归检查。

Bug 可以在合理假设下推进，但必须标明假设；假设影响产品行为或无法验证时必须升级确认。

### 页面或体验任务

必须补充：

- 用户参与场景。
- 页面必须展示的信息。
- 用户必须能完成的动作。
- 阻塞、分歧、风险确认如何可见。
- Review、门禁、复盘如何可见。
- MiniMax 参与方案或 Review 的安排。

### Review 任务

必须补充：

- Review 对象。
- Review 视角。
- 必须检查的问题。
- 不在本次 Review 范围内的问题。
- 输出格式。
- 是否允许直接修改。

Review 方不得把 Review 任务扩展为重写实现，除非启动包明确允许。

### 质量门禁或交付检查任务

必须补充：

- 需要检查的门禁项。
- 允许通过的标准。
- 失败时阻断范围。
- 是否允许修复。
- 是否涉及 Git 写入或 feature 分支推送。

### Dogfooding 或复盘任务

必须补充：

- 复盘对象。
- 需要统计或观察的指标。
- 过程问题。
- 产品改进建议。
- 架构执行建议。
- 流程改进建议。
- 哪些建议需要产品负责人确认后才能进入基线。

## 零阶段手工启动包示例

```text
任务 ID：T2
任务类型：功能需求 / 页面入口开发
当前身份：Codex
任务来源：docs/execution/15-implementation-plan.md
关联工作项：P0-14 / P0-15 / P0-16 页面变更补齐
优先级：P0

任务目标：
补齐最小页面工作项录入入口，让用户可以通过页面提交功能需求或 Bug 修复，并真实创建 WorkItem。

范围内事项：
- 页面录入入口。
- 功能需求 / Bug 修复类型选择或识别。
- 创建 WorkItem 初始记录。
- 创建后展示最小初始状态。

范围外事项：
- 完整统一聊天室。
- 完整工作项详情页。
- 完整 Review / 门禁 / 复盘视图。
- 高保真视觉设计。

交付物：
- 页面录入入口。
- 相关验证记录。
- 必要文档更新。

必读文档：
- docs/00-index.md
- docs/product/14-page-user-participation-proposal.md
- docs/product/15-page-change-implementation-clarifications.md
- docs/architecture/16-page-change-architecture-clarifications.md
- docs/execution/15-implementation-plan.md
- docs/agents/05-agent-codex.md

依赖：
- T1 已完成。
- T3 持久化出口或其上层应用服务可用于创建 WorkItem。

允许动作级别：
- 文件修改。
- 检查执行。
- 提交准备。

文件/模块边界：
- 仅允许修改 T2 页面录入入口、WorkItem 创建调用路径、必要的测试/验证文件和相关执行文档。
- 不得修改完整 T13 页面主界面、状态机、A2A 编排、Review/门禁/复盘完整视图，除非启动包另行授权。

维护性注释要求：
- 对非显而易见的创建流程、类型识别/选择逻辑、页面与统一持久化出口的连接点补充必要注释。
- 不要求逐行注释，不用显而易见注释堆数量。

Git 身份要求：
- 若发生 Git 写入动作，使用当前执行身份对应的 `Clowder Codex <codex@clowder.local>`，除非任务启动上下文明确覆盖。

worktree / 分支要求：
- 若与其他 Agent 并行开发，应在独立分支或 worktree 中执行，并记录任务、分支/worktree、负责人和冲突状态。

Review 方：
Claude

MiniMax 参与要求：
T2 可暂不强制 MiniMax 参与，但必须记录该临时调整；MiniMax 必须在 T13 或 T16 前参与页面体验 Review。

验收标准：
- 用户可以通过页面录入工作项。
- 页面入口真实创建 WorkItem。
- 创建后可看到 ID、类型、当前状态、标题或目标、低置信度或待确认提示。
- CLI 不作为本任务的新基线验收入口。

质量门禁：
- 页面录入路径验证通过。
- WorkItem 创建结果可读取。
- 相关自动检查或手动验证记录已归档。
- 非作者 Review 完成后才能视为通过。

失败处理：
- 如果页面入口不能真实创建 WorkItem，任务不得通过。
- 如果页面入口和 CLI 内部入口仍写入不同事实来源，必须记录为进入 T5 前阻塞。
- 如果 MiniMax 仍未参与，必须记录为 T13 或 T16 前待触发体验 Review。

输出记录位置：
- 优先写入 T2 结果文档或当前执行状态看板指向的独立执行记录。
- Review 结果写入对应 Review 文档或执行记录。

建议下一状态或交接状态：
- 页面入口通过 Review 后，交接给 T5 前 WorkItem 单一事实来源确认。
- 未通过 Review 时，保持在待修复或待 Review 状态。

后续阻断关系：
- 若 WorkItem 事实来源未统一，阻断 T5 状态机启动。
- 若 T2 页面录入不成立，阻断 T13 完整页面主界面和 T16 E2E 验收。

禁止事项：
- 不得把全部页面入口推迟到 T13。
- 不得把 CLI 输出包装成页面主入口。
- 不得扩大到完整 T13 范围。
- 不得部署。
- 不得合并主干。
- 不得绕过非作者 Review 或质量门禁。
```

## 待架构侧落地

以下事项属于架构执行设计，不再阻塞本产品侧模板使用：

- 任务启动包在系统中的承载形态。
- Harness 如何生成、校验、下发和记录任务启动包。
- 哪些字段需要结构化存储，哪些可以作为文本上下文。
- 任务启动包如何与状态机、A2A 事件、Review、质量门禁、worktree 和 Git 记录关联。
- 零阶段人工启动包如何平滑迁移到自动 Harness 启动包。
